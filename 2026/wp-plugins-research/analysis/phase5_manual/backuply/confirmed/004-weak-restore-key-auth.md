# BACKUPLY-004: Weak Custom Token Auth in Restore Flow

## Severity: MEDIUM (CVSS ~6.5)

## Summary

The restore flow uses a custom authentication scheme (`RESTORE_KEY` and `status_key`) instead of WordPress nonces. These tokens are static secrets stored in predictable file locations and the database, creating multiple exposure vectors.

## Authentication Architecture

### RESTORE_KEY
- Generated by `backuply_csrf_get_token()` = `bin2hex(openssl_random_pseudo_bytes(32))` (64 hex chars)
- Stored in WP option `backuply_config_keys`
- Also written to file `wp-content/backuply/backuply_config.php` (prefixed with `<?php exit();?>`)
- Used by: `backuply_verify_self()`, `restore_ins.php`, `backuply_creating_session`, `backuply_restore_response`, `backuply_update_serialization_ajax`

### status_key
- Generated by `wp_generate_password(32, false)` (32 alphanumeric chars)
- Stored in file `wp-content/backuply/status_key.php` (prefixed with `<?php exit();?>`)
- Used by: `backuply_verify_status_log()`, `status_logs.php`
- Additional check: `status.lock` file must exist and be modified within last 30 minutes

## Weaknesses

### 1. Non-Rotating Tokens

The RESTORE_KEY is regenerated on plugin update (`backuply_update_check`) and when `backuply_get_restore_key` is called. But during the 10-minute restore window, the key is static and reusable across multiple requests. WordPress nonces, by contrast, are tied to user sessions and action context.

### 2. Token Returned to Frontend

**File:** `main/ajax.php`, lines 536-558:
```php
function backuply_get_restore_key(){
    backuply_ajax_nonce_verify();
    backuply_update_restore_key();
    $config = backuply_get_config();
    wp_send_json(array('success' => true, 'restore_key' => $config['RESTORE_KEY'], 'backuply_key' => $config['BACKUPLY_KEY']));
}
```

Both the RESTORE_KEY and BACKUPLY_KEY are sent to the admin's browser. Any XSS vulnerability on the admin page (even in another plugin) during a restore operation could steal these keys.

### 3. File-Based Storage on Shared Hosting

The config file `backuply_config.php` is protected by:
- `<?php exit();?>` prefix (prevents PHP execution of the JSON data)
- `chmod 0600`
- `.htaccess` deny (Apache only)

On Nginx, there is no equivalent of `.htaccess`. The `<?php exit();?>` prefix prevents direct PHP access, but if the web server serves it as plaintext (misconfigured MIME types, backup of file with different extension), the key is exposed.

### 4. Timing-Safe Comparison Missing

**File:** `restore_ins.php`, line 172:
```php
if(urldecode($_REQUEST['restore_key']) == $config['RESTORE_KEY']) {
```

Uses loose comparison (`==`) instead of `hash_equals()`. While timing attacks against remote HTTP are generally impractical, this is still a security anti-pattern.

### 5. Both Keys Returned Together

The `backuply_get_restore_key` endpoint returns both `RESTORE_KEY` and `BACKUPLY_KEY`. A leak of this single response compromises both authentication paths.

## Impact

- Token leakage through XSS enables the full BACKUPLY-001 privilege escalation chain
- Shared hosting environments increase file-level exposure risk
- The custom auth scheme bypasses all WordPress security hardening (brute-force protection, session management, etc.)

## Classification

- CWE-798: Use of Hard-Coded Credentials (static token auth instead of session-based)
- CWE-522: Insufficiently Protected Credentials
- CWE-208: Observable Timing Discrepancy (minor)

## Recommendation

1. Replace custom token auth with WordPress nonces for all internal AJAX calls
2. For the async restore flow (where WP session may be invalid), use single-use tokens with `wp_hash()` + `wp_salt()`
3. Use `hash_equals()` for all token comparisons
4. Never return both authentication keys in a single response
