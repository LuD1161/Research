# WPVIVID-002: Backup Files Directly Accessible on Non-Apache Servers

## Severity: MEDIUM (CVSS ~5.3)

## Summary

WPvivid stores backup archives (containing full database dumps and site files) in a well-known, web-accessible directory (`wp-content/wpvividbackups/`) protected only by `.htaccess` rules. On Nginx, LiteSpeed, or any non-Apache server, these files are directly downloadable by unauthenticated users if the filename can be guessed.

## Vulnerable Code

### Backup Directory Protection (wpvivid-backuprestore.php, lines 118-148)

On plugin activation, the plugin creates `.htaccess` and `index.html` files:

```php
$backup_dir = $dir['path'];  // default: "wpvividbackups"
// ...
mkdir(WP_CONTENT_DIR . DIRECTORY_SEPARATOR . $backup_dir, 0777, true);
fopen(WP_CONTENT_DIR . '/' . $backup_dir . '/index.html', 'x');
$tempfile = @fopen(WP_CONTENT_DIR . '/' . $backup_dir . '/.htaccess', 'x');
if($tempfile) {
    $text = "<IfModule mod_rewrite.c>\r\nRewriteEngine On\r\nRewriteRule .* - [F,L]\r\n</IfModule>";
    fwrite($tempfile, $text);
}
```

### Backup Filename Generation

Backup filenames follow the pattern:
```
wpvivid-{uniqid}_{YYYY-MM-DD-HH-MM}_backup_{type}.zip
```

Where `{uniqid}` is generated by PHP's `uniqid('wpvivid-')`, which produces a hex string based on the current microsecond timestamp. Example: `wpvivid-65a3f2b0c1d4e_2024-01-15-03-30_backup_all.zip`

## Attack Scenario

### Prerequisites
- Target site runs on Nginx (or any non-Apache web server)
- A backup exists in the default backup directory

### Steps
1. Attacker identifies that the target uses WPvivid (e.g., via plugin fingerprinting)
2. Determine approximate backup time (scheduled backups often run at predictable times)
3. Brute-force the `uniqid` portion of the filename
4. Request `https://target.com/wp-content/wpvividbackups/{guessed_filename}`

### Brute-force Feasibility

`uniqid()` returns a 13-character hex string representing microseconds since epoch. If the attacker knows the backup ran within a 1-hour window:
- 1 hour = 3,600,000,000 microseconds
- Each guess is a simple HTTP HEAD request
- At 1000 requests/second: ~1 hour to enumerate
- With scheduled backups (known to minute precision): only ~60,000,000 values, feasible in minutes

## Impact

- **Full database exposure**: Backup ZIP contains SQL dumps with all user credentials, emails, and site data
- **Source code exposure**: All PHP files, wp-config.php with database credentials, API keys, salts
- **File system exposure**: Uploaded media, private documents

## Mitigating Factors

- Only affects non-Apache servers (Nginx, IIS, etc.)
- Requires guessing the backup filename (feasible but not trivial)
- Some hosting providers add blanket rules blocking direct access to wp-content subdirectories
- `index.html` prevents directory listing (but does not prevent direct file access)

## Remediation

1. Add `Content-Disposition` download restriction via PHP-mediated downloads only
2. Store backups outside the web root
3. Use cryptographically random filenames (e.g., `wp_generate_password(32, false)`)
4. Add server-agnostic protection (e.g., check for a valid download token in a PHP handler)
